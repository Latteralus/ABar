## 3. Mental model of the codebase

**The simulation engine is not React.** `SimulationEngine` (`src/simulation/engine/SimulationEngine.ts`) owns one mutable `GameState` object and a `setInterval`-driven clock. It mutates that object directly (push to arrays, flip fields) — no immutability library, no Redux. React never touches this object; it only ever reads snapshots.

**The snapshot bridge is `stores/gameStore.ts` (zustand).** Every time the engine finishes a tick (or a player command runs), it calls back into the store with `set({ state: { ...engineState } })` — a **new top-level object**, but the nested arrays/objects inside it keep the same references unless the engine itself reassigned them. This means:

> **Rule for any new component:** select the whole state once — `const state = useGameStore(s => s.state)` — and read fields off it during render. Do **not** write a zustand selector that narrows into a nested array (e.g. `useGameStore(s => s.state.customers)`) — because that array's reference often doesn't change between commits even though its contents did, so a narrow selector can silently miss updates. Every screen in `src/features/` already follows the "select whole state" pattern; copy that.

**`GameState` is now per-property, not flat (Real Estate/Neighborhoods).** Everything location-specific — `employees`, `customers`, `customerGroups`, `inventory`, `purchaseOrders`, `equipment`, `attractions`, `menu`, `barCleanliness`, `tabs`, `receipts`, `tasks`, `orders`, `reputation`, `reviews`, `activePromotions`, `dailyReports`, and most `bills` — lives on `OwnedPropertyState` (`types/property.ts`), one per owned/leased location, inside `GameState.properties: OwnedPropertyState[]`. `GameState.activePropertyId` names which one is currently live-simulated. **Never read those fields off `GameState` directly** — use `activeProperty(state)` (`simulation/engine/activeProperty.ts`) to get the live one, or `findOwnedProperty(state, propertyId)`/`backgroundProperties(state)` for a specific/every-other one. Almost every engine function that used to take just `(state: GameState, ...)` now also takes a `(prop: OwnedPropertyState, ...)` parameter — `SimulationEngine.advanceOneMinute()` resolves `activeProperty(state)` once per tick and threads it through every call. `GameState.bills` only holds the startup loan (genuinely company-wide, one per save); every other bill kind lives on `prop.bills` — `finance.ts`'s `outstandingBills`/`outstandingBillTotal`/`markOverdueBills` now take a plain `Bill[]` array so callers can pass either or combine both. `LedgerEntry` gained an optional `propertyId` so ledger entries can be attributed and a daily report can be scoped to just its own property's entries (`dayCycle.buildDailyReport` filters `state.ledger` down to `propertyId === prop.propertyId || undefined` before summarizing — otherwise a background property's estimated revenue would leak into the active property's real report). `dayCycle.openDay`/`closeDay` still take just `(state, bus)` — they resolve `activeProperty`/`backgroundProperties` internally; `openDay`'s customer/task/attraction reset is scoped to the active property only, but purchase-order delivery/contract-repair resolution/bill-overdue-marking loop over *every* owned property since those are deterministic and already-scheduled regardless of which property is live. `closeDay` similarly runs payroll/weekly-bills for every property, but only the active one gets a real `dailyReports` entry — every background property instead gets `backgroundOperations.runBackgroundPropertyDay` (posts an estimated revenue/COGS pair from `computeBackgroundEstimateProfile`'s trailing-7-day average, plus a proportional inventory drawdown), which explicitly never touches `dailyReports` since that array is the real history the estimate is computed from.

**Player actions vs. simulation ticks are two different code paths that both mutate the same `GameState`:**
- Tick path: `SimulationEngine.advanceOneMinute()` → `customerArrivals` → `customerLifecycle` → `orderProcessing.ensureOrderTasks` → `seatingTasks.ensureSeatingTasks` → `cleaning.ensureCleaningTasks` → `intoxicationHandling.processIntoxicatedCustomers` → `employeeAI` (assigns/advances tasks, dispatches completion handlers that call into `orderProcessing`/`payments`) → closing check → `dayCycle`. All the `ensure*` functions follow the same pattern: scan state, and if a condition needs a task that doesn't already exist (checked via `t.status !== "complete" && t.status !== "cancelled"`), push a new `ServiceTask`. Copy that pattern for new task-driven systems rather than inventing a new one.
- Command path: UI calls a `useGameStore` action → that calls a pure function in `services/commandService.ts` (validates, mutates state, returns `{success, error?}`) → the store calls `engine.commitNow()` to push a fresh snapshot and usually `saveService.save(...)`.

**Task role-eligibility is centralized.** `simulation/tasks/roleEligibility.ts` maps every `TaskType` to the `EmployeeRole[]` allowed to do it (including cross-role coverage, e.g. barbacks can `deliver_drink`). Task-creation call sites pull from `rolesFor(taskType)` instead of hardcoding role arrays — when Stage 3+ adds new task types (`prepare_food`, `wash_dishes`, `repair_equipment`, ...), add them here first.

**Personality effects are centralized too.** `simulation/engine/personalityEffects.ts` folds an employee's `personality: PersonalityTrait[]` into four numbers (`personalitySpeedMultiplier`, `personalityWasteMultiplier`, `personalitySatisfactionBonus`, `hasCalmBonus`), combining multiple traits multiplicatively (speed/waste) or additively (satisfaction). The per-trait effect table lives in `config/personalityConfig.ts`. If you add a new trait to `types/employee.ts`'s `PersonalityTrait` union, you must add a matching entry to `PERSONALITY_EFFECTS` or it silently does nothing.

**Food shares the drink task pipeline instead of duplicating it.** `employeeAI.ts`'s `handlePrepareItemComplete`/`handleDeliverItemComplete` (formerly `handlePrepareDrinkComplete`/`handleDeliverDrinkComplete`) branch on `product.category === "food"` only where behavior actually differs (which task type/role/skill to queue next; whether to bump intoxication on delivery) — everything else (`consumeInventoryForOrder`, ledger posting, tab line items, satisfaction math) was already category-agnostic and needed zero changes. If you add a third preparable category later, extend these branches rather than forking new handler functions.

**Equipment ownership lives on each `OwnedPropertyState.equipment`, not on the catalog `Property`.** A catalog entry's `Property.existingEquipment` (`data/properties/*.ts`) is only the *starting* loadout for that location, copied in fresh by `propertyStateFactory.createOwnedPropertyState` — used both for the very first property at new-game time and for every later `commandService.leaseOrBuyProperty`, so both paths start a location identically. Purchases (`commandService.purchaseEquipment`) push onto `activeProperty(state).equipment` directly. A recipe's `Recipe.requiredEquipmentCategory` is checked via `orderProcessing.hasRequiredEquipment(prop, productId)` — reuse that function rather than re-deriving the check; it's called both as a menu-activation gate (`commandService.setMenuActive`) and as a defensive backstop inside `selectProductForCustomer`.

**Equipment health is centralized in `simulation/engine/equipmentMaintenance.ts`.** `Equipment.currentStatus` follows the exact state machine from Master Plan §48 (`operational → degraded → failed → awaiting_repair → under_repair → operational`). `isEquipmentUsable(e)` (true only for `operational`/`degraded`) is the one predicate everything else is built on: `orderProcessing.hasRequiredEquipment` uses it to gate ordering/menu-activation, `getStorageUsage` (spoilage.ts) uses it to decide whether a fridge/freezer's capacity counts. Performance degradation (not full failure) is a multiplier, not a gate: `equipmentSpeedMultiplier`/`equipmentWasteMultiplier` scale with how far the *healthiest usable unit* in a category has fallen below the degraded threshold — a second healthy unit of the same category fully masks a broken one. Two repair paths exist as parallel player choices, not a fallback chain: an on-staff `maintenance` employee gets an auto-queued `repair_equipment` task the moment something fails (`ensureMaintenanceTasks`, same shape as `cleaning.ensureCleaningTasks`); contract repair is a player command (`commandService.requestContractRepair`) that resolves at a later day's `openDay()` (mirrors purchase-order delivery timing) rather than tracking an exact in-day minute.

**Spoilage ages an item's whole pool, not individual batches.** `InventoryItem.daysSinceLastRestock` is a single counter per item, incremented once per `closeDay` and reset to 0 on delivery (`dayCycle.deliverDuePurchaseOrders`). `simulation/engine/spoilage.ts`'s `applySpoilage` spoils an item's *entire* remaining `quantityOnHand` the moment `daysSinceLastRestock` reaches its effective shelf life — there's no FIFO/batch tracking, matching the existing single-`averageUnitCost`-pool simplification. Storage capacity (`getStorageUsage`, also in `spoilage.ts`) doesn't block purchases; being over capacity for an item's pool (general/refrigerated/frozen) instead halves its effective shelf life (`SPOILAGE_CONFIG.improperStorageShelfLifeMultiplier`) — that's the entire mechanism by which storage capacity "matters" in Stage 3.

**Attractions are their own domain, not bolted onto Equipment or Customer.** `types/attraction.ts` defines `Attraction`/`AttractionQueueEntry`/`AttractionSession` independently — condition/breakdown mirrors `equipmentMaintenance.ts`'s state machine in *shape* only (`operational → degraded → failed → awaiting_repair → under_repair`), via its own `simulation/engine/attractionCondition.ts`. Nine focused modules (`attractionQueue`, `attractionSessions`, `attractionCondition`, `attractionTasks`, `attractionRevenue`, `attractionReporting`, `customerAttractionDecisions`, plus `data/attractions/attractionCatalog.ts` and `config/attractionConfig.ts`) split the concerns the same way Stage 3/4 split food and equipment work. Two new `CustomerStatus` values (`waiting_for_attraction`, `using_attraction`) do all the "can't do two things at once" locking — no new `Customer` field points at "which attraction" (queue/session lookups just scan the small `state.attractions` array by customer ID; see `attractionQueue.findAttractionForCustomer`). The one subtlety: the existing order pipeline (`take_order`→`prepare_X`→`deliver_X`) doesn't care what status a customer is "supposed" to be in, so a participant *can* order a drink while queued/playing for free — except `employeeAI.handleTakeOrderComplete`/`handleDeliverItemComplete` both have a one-line guard so they don't stomp `using_attraction`/`waiting_for_attraction` back to the normal `waiting_for_drink`/`consuming` statuses. If you touch either handler, keep that guard.

**A queue entry is a decision-unit (a whole arrival group, or one solo customer), never a split group,** and session formation (`attractionQueue.ensureAttractionQueueProgress` → `selectNextSession`) prefers the first *single* queue entry that already fits `[minParticipants, maxParticipants]` alone, falling back to combining consecutive small entries FIFO only when nothing qualifies alone. Greedily combining strict front-to-back would let an unpaired solo customer at the front permanently block a full group of 4 arriving behind them — this was a real bug caught by `tests/attractions.test.ts` during development, not a hypothetical. Pool's own `minParticipants` is 1 (solo play is allowed — see §1's bug-fix note), so in practice every queue entry always qualifies alone and the combining fallback is currently unreachable for pool specifically; it's still exercised directly in `tests/attractions.test.ts` via `selectNextSession(attraction, 2, 4)` so the logic stays correct for a future attraction type that genuinely needs 2+ players.

**Determinism:** all gameplay-affecting randomness goes through `SeededRandom` (`simulation/random/SeededRandom.ts`, mulberry32, single 32-bit state). The engine keeps one instance; `state.rngSeed`/`state.rngState` are persisted so reload continues the same sequence. Player-triggered randomness (e.g. generating hiring candidates) routes through `engine.generateWithRandom(fn)` so it uses the *same* RNG instance, not a fresh one. Entity IDs (`services/idService.ts`) use `nanoid` — intentionally **not** routed through SeededRandom, because IDs are opaque labels, not gameplay decisions (documented in that file).

**Ledger is the source of truth for money.** `types/financial.ts` defines the full chart-of-accounts categories (revenue/cogs/opex/asset/liability/equity) even though Stage 1 only posts a subset. Daily/weekly summaries are derived from the ledger on demand via `simulation/engine/ledgerSummary.ts` (`summarizeDay`, `summarizeRange`) — never stored as pre-computed totals. Reuse those functions rather than re-deriving revenue/profit math in a new screen.

**Every `state.cash` mutation must post a paired `asset_cash` ledger entry** (credit = increase, debit = decrease — that's this codebase's own sign convention for the category, not a GAAP claim). `simulation/engine/cashFlow.ts`'s `cashAsOfDay`/`summarizeCashFlow` depend on this being universally true to reconstruct historical cash balances and classify cash flow into operating/investing/financing — an audit found two commands that had drifted from this (contract repairs, initial game-start funding), both fixed. If you add a new command or engine function that changes `state.cash`, post the matching `asset_cash` entry in the same place, or the Financials screen's Cash Flow card will silently under/over-count.

**Never check a customer's spending against raw `Customer.totalSpent`.** It only updates when a tab is actually paid (`payments.closeTabAndPay`), so it's always stale/zero while a tab is open. Any affordability or "can they afford another round" check must go through `payments.customerSpentSoFar(prop, customer)` instead (paid history + the currently-open tab's running subtotal) — this was a real bug (customers could out-order their budget indefinitely) until fixed. If you add a new spending gate, use that helper.

**Customers are not purged mid-day.** A customer who leaves gets `status: "left"` but stays in `state.customers` until `dayCycle.openDay()` clears the roster for the next day. This is what lets `buildDailyReport` compute accurate counts/satisfaction/loss-reasons at close. If you change this, the daily report generator breaks.

**Paying is not the same as leaving.** When `process_payment` completes, the customer's tab is closed but their `status` becomes `"leaving"`, not `"left"` — they still count as present/occupying a seat. `customerLifecycle`'s `"leaving"` case waits `CUSTOMER_BEHAVIOR_CONFIG.departureLingerMinutes` before actually calling `departCustomer(...)` to flip them to `"left"`. This was a real reported bug (customer count dropping to 0 the instant someone paid) — if you touch `handleProcessPaymentComplete` in `employeeAI.ts`, keep this two-step handoff intact. There's a regression test for it in `tests/customerLifecycle.test.ts`.

**A side state machine tracks intoxication removal separately from the main customer status.** `Customer.removalStage` (`"warned" | "police_called"`, undefined = not in the removal flow) and `Customer.removalStageEnteredAtGameMinute` are driven entirely by `simulation/engine/intoxicationHandling.ts` and don't interact with `Customer.status`'s normal lifecycle — a customer can be `"consuming"` and mid-removal-flow simultaneously. If security is on staff, this side machine is bypassed entirely in favor of a real `remove_customer` task.

## 5. Where the money/units live (easy to get wrong)

- **All currency is integer cents** (`Cents` type alias = `number`). Never store dollars as floats. `utils/money.ts` has `formatCents`/`dollarsToCents`.
- **Game time** is minutes-since-open (`GameState.gameMinute`, 0–720), *not* absolute minutes since save start. `GameState.gameDay` is the separate day counter. `simulation/clock/gameTime.ts` converts minute-of-day to a wall clock (opens 2:00 p.m., wraps through midnight, closes 2:00 a.m. at minute 720).
- **Recipe quantities** are in whatever unit the ingredient's `InventoryItem.unit` is (ounces, servings, bottles...) — see `data/recipes/recipes.ts` and `data/products/inventoryCatalog.ts`.

## 6. Deliberate simplifications (so you don't "fix" them by accident)

- **No immer/structural sharing library.** The engine mutates in place; the store re-wraps at the top level each commit. This is why the "select whole state" rule in §3 matters. Adding immer later is possible but would touch every engine module.
- **ESLint is now configured** (`eslint.config.js`, flat config — `typescript-eslint` recommended rules, non-type-checked for speed, plus `eslint-plugin-react-hooks`/`eslint-plugin-react-refresh`). `npm run lint` runs it. Deliberately not type-aware (no `recommendedTypeChecked`) to keep it fast and low-noise for a first pass; revisit if the project wants stricter type-based rules later. `no-undef` is off for `.ts`/`.tsx` (TypeScript itself already catches undefined references, and the rule doesn't understand TS-only constructs). Two pre-existing `react-refresh/only-export-components` warnings remain (`CustomerTable.tsx`'s `STATUS_LABEL`, `FloorView.tsx`'s `deriveFloorLayout` — both export a non-component alongside a component); left alone since fixing them means relocating those exports to new files, a layout call left for whoever touches those files next. **Still no Prettier config** — not addressed by this pass.
- **Tips are still simplified, but the bar share is now player-adjustable.** `GameState.policies.barTipSharePercent` controls the split. The bar's share is posted to the ledger (`revenue_bar_tip_share`); the employee share is split evenly across `state.employees` and tracked as `Employee.performance.tipsEarnedCents` (career cumulative, not day-scoped, not run through the ledger as a payable liability). Full tip payout/payable accounting is still not modeled.
- **Inventory purchases post as a plain `asset_cash` debit**, not an `opex_*` category — intentional, because buying stock is a balance-sheet asset swap (cash → inventory), not an expense. It only becomes COGS when a recipe actually consumes it (see `orderProcessing.consumeInventoryForOrder` → `cogsCategoryForProduct`).
- **Auto-open** is implemented as "immediately call `openDay()` again right after `closeDay()` if `autoOpenEnabled`" — there's no real-time between-day clock, since the between-day phase is player-paced, not simulated in real time.
- **Payroll is accrued once per closed day, not continuously by the minute.** `Employee.wagePerShiftCents` is posted daily by `finance.accrueDailyPayroll`, then rolled into weekly payroll bills. This is intentionally simpler than shift-by-shift labor costing.
- **Debug panel** (`features/settings/DebugPanel.tsx`) is gated by `import.meta.env.DEV`. It currently has 9 tools: add/remove cash, skip to near-closing, force a spoilage check, fail all equipment, fail all attractions, trigger Sunday billing now, force negative cash, and force bankruptcy now. Extend it per Master Plan §51 as needed — it's intentionally minimal, not exhaustive.
- **Skill progression uses a lifetime-average activity factor, not "tasks completed today."** `skillProgression.applyShiftProgression` approximates how busy an employee typically is from `performance.* / shiftsWorked` rather than snapshotting daily task counts (which would need a new persisted "counts as of day start" structure). An employee who was very busy on day 1 and idle every day since will still show as "experienced" in the growth rate — acceptable, but not literally per-day activity as Master Plan §20 describes.
- **Restocking (barback moving stock from storage to bar-accessible stock) was deliberately *not* built in Stage 2**, even though barbacks are hirable and "Restock bar" is a task type named in Master Plan §25. The current `InventoryItem` model has one `quantityOnHand` per item, not a separate storage-pool vs. bar-accessible-pool split — without that split, a "restock" task would have nothing real to move and would just be a fake animation. Build the storage split first if you want real restocking mechanics.
- **Cooks now have real work (Stage 3); dishwashers still don't.** Cooks handle `prepare_food` tasks. Dishwashing mechanics (Stage 3's `data/products/inventoryCatalog.ts` didn't need glassware/dish tracking) still don't exist, so dishwashers remain hired-but-idle until that's built.
- **Spoilage has no batch/lot tracking.** An item's whole `quantityOnHand` pool shares one `daysSinceLastRestock` clock and spoils all at once when it expires — see §3. Building true FIFO batches would require reworking the single-`averageUnitCost`-pool inventory model, which Stage 3 deliberately left alone.
- **Storage capacity never blocks a purchase.** The existing catalog's `reorderTarget`s (set in Stage 1/2) already exceed `Property.storageCapacityUnits` and the starter fridge's capacity if fully stocked — a hard block would have meant rebalancing pre-existing numbers, out of scope for "food and spoilage." Over-capacity instead just halves effective shelf life for perishables in that pool (§3) and shows a warning badge on the Inventory screen.
- **The barback-restocking storage-pool split described in an earlier version of this doc was *not* built.** Stage 3 added per-pool storage *capacity* tracking (general/refrigerated/frozen, via `getStorageUsage`), but each `InventoryItem` still has a single `quantityOnHand` — there's still no separate "in general storage" vs. "bar-accessible" split, so barback `clean_bar`-style restocking tasks still wouldn't have anything real to move. That remains open for whoever tackles it.
- **Manager is the only role still not hirable** (`HireEmployeeModal.tsx`'s `HIRABLE_ROLES` excludes it) — manager automation is Stage 8. It already has wage config in `EMPLOYEE_ROLE_CONFIG` for when it's needed.
- **`breakdownRisk` and `energyUsage` from Master Plan §33's equipment stat list are still not stored fields.** Breakdown risk is a formula derived from `condition` at read time (`equipmentMaintenance.ts`'s internal `degradedSeverity`), not a data point that needs persisting. Utilities are now billed in Stage 5, but they are still a flat weekly bill rather than a per-equipment energy calculation.
- **Equipment wear only has task hooks for `bar_station`/`cooking_equipment`/`point_of_sale`.** Every owned item gets the daily time-based decay (`applyDailyEquipmentWear`), but only those three categories get *extra* wear from actually being used, because those are the only categories with a task type that completes today. `draft_system`/`glass_washer`/`dishwasher`/`storage_shelving`/`security_system`/`maintenance_tool` and other expanded catalog items still break and can be repaired (the state machine doesn't care why something's degraded), but most do not yet gate task flow — same "exists as data, no gameplay hook yet" pattern as other stages.
- **Equipment-space limits are hard purchase blocks; storage capacity is still a spoilage-risk lever.** `Property.equipmentFloorSpaceUnits` limits equipment purchases through `commandService.purchaseEquipment`; if a new item would exceed that cap, the purchase is rejected. This is intentionally stricter than inventory storage overflow, which remains allowed and only increases spoilage risk.
- **Contract repair resolves at a day boundary, not a precise in-day minute.** `Equipment.contractRepairDueGameDay` is checked in `dayCycle.openDay` (mirrors `deliverDuePurchaseOrders`'s "arrives at next open" timing) rather than tracking exact game-minutes across a potential midnight rollover — Master Plan §34 doesn't require sub-day precision here, and this sidesteps a whole class of cross-day minute-arithmetic bugs for something that doesn't need it.
- **Menu margin is a derived UI convenience, not a stored field.** `MenuListing` still only has `price`. The Menu & Pricing screen computes `margin% = (price - cost) / price` for display and, on edit, solves back for `price = cost / (1 - margin/100)` and calls the existing `setMenuPrice` command — no schema change, no new command.
- **Purchasing's "Max Capacity"/"Buy Max" only account for current on-hand stock, not other line items in the same draft order.** If you use Buy Max on two items sharing the same storage pool in one order, both will independently fill to "room left based on current on-hand," which can jointly overshoot the pool once both deliver. This is a display/convenience feature, not a hard constraint (storage capacity has never hard-blocked purchases — see the spoilage-penalty simplification below), so the overshoot just means a bit more spoilage risk, not a broken invariant.
- **Four attraction types are implemented (pool table, darts, arcade cabinet, karaoke booth).** The architecture (queue/session/condition/tasks/revenue/reporting, all keyed by `AttractionCategory`) is generic on purpose — nothing branches on a specific category inside the generic modules, so a fifth type is still just a catalog addition, not a rewrite.
- **An oversized arrival group (bigger than `maxParticipants`) never splits to fit a table**, and a group larger than the table's max simply never becomes queue-eligible for it. Splitting a group across two games (some play, some watch/wait) was judged more complexity than the feature warrants for a first attraction type.
- **Session formation prefers a single queue entry that already fits `[min, max]` over combining an earlier smaller entry with it.** It does *not* consider "solo customer + the group right behind them, combined, would also fit" — that qualifying entry just goes alone and the solo customer keeps waiting for another small party. Documented in `attractionQueue.selectNextSession`'s comment; full bin-packing was judged not worth it for a 2-4-seat table.
- **Attraction floor space is a hard purchase block; storage capacity is only ever a soft one.** These are deliberately different rules for different reasons: physical floor space genuinely can't be worked around (see `Property.attractionFloorSpaceUnits`, checked in `commandService.purchaseAttraction`), whereas storage overflow is a risk lever (Stage 3), not a hard wall.
- **Attraction queues/sessions are blunt-reset at `openDay`, not gracefully unwound.** Since `state.customers` is wiped every day anyway, there's nothing left to notify — `attraction.queue = []` and `attraction.activeSession = null` mirror the existing `state.tasks = []`/`state.orders = []` reset rather than trying to log a per-party "sorry, closing" message.
- **"Estimated secondary sales" is exactly that — an estimate, not a reconciled figure.** `Attraction.estimatedSecondarySalesCents` is incremented by a chosen listing's price the moment a participant's "order while playing" `take_order` task is created (`employeeAI.handleTakeOrderComplete`), not tied to whether that order is later cancelled for missing inventory or discounted — same spirit as `wasteCostCents`/other running counters elsewhere in the codebase that are informative, not audited.
- **No customer "personality" system was added for attraction interest.** The request asked for interest to vary by "archetype, personality, group size, price sensitivity, and current visit state" — this codebase only has a personality system for *employees* (`PersonalityTrait[]`), so `Customer.attractionAffinity` (a new per-archetype 0-100 field, like `priceSensitivity`/`patience`) is the customer-side equivalent rather than building a parallel trait system just for this.
- **A background property's "day" is a single estimated revenue/COGS number, not a scaled-down simulation.** `backgroundOperations.runBackgroundPropertyDay` doesn't spawn any customers, employees don't do anything, and no tasks run for a background property — it's a deliberate simplification (per the user's own request: "calculates the last few days costs/income... and uses those values") rather than a lightweight parallel simulation. The trailing average only updates when the property is *active*; while background, the same `backgroundEstimate` snapshot (taken the moment you switch away, see §3) repeats unchanged day after day until you switch back and generate new real history.
- **Property acquisition/lease pricing and the liquidation/resale percentages are static config, not negotiable or dynamic.** `REAL_ESTATE_CONFIG.equipmentAttractionResalePercent`/`propertyResalePercent` are flat 50%/65% regardless of equipment condition, how long you've owned it, or market conditions — there's no haggling, no lease-term negotiation, and rent doesn't fluctuate over time for a given property.
- **The property catalog is a fixed 4-entry ladder, not a randomized/expanding market.** `data/properties/index.ts`'s `PROPERTY_CATALOG` never changes at runtime — every property is always available to lease/buy (once not already owned) from game start; there's no "new listing becomes available" event or limited-time availability.
- **Neighborhood `competitionLevel` is a static per-property stat baked into the demand formula, not a dynamic competitor system.** It's conceptually distinct from the still-unbuilt Stage 7 "Market Simulation" (1-4 background competitors that would presumably react to the player) — don't confuse the two when picking up Stage 7.

## 9. If something looks broken

- **Type error mentioning `node:path` / `__dirname` in `vite.config.ts` or `vitest.config.ts`:** don't reintroduce those — they were replaced with `fileURLToPath(new URL(...))` specifically to avoid needing CJS globals in an ESM config. `@types/node` is a devDependency now, so plain Node built-ins do resolve, but keep the ESM-style path resolution already in place.
- **Stray `vite.config.js` / `.d.ts` / `tsconfig.tsbuildinfo` appearing at repo root after a build:** `tsconfig.node.json` has `tsBuildInfoFile`/`outDir` pointed at `node_modules/.tmp/...` specifically to prevent this. If it happens again, something changed that config — check it didn't regress to emitting into the project root. `.gitignore` already excludes `*.tsbuildinfo`.
- **A screen isn't re-rendering when engine state clearly changed:** you're almost certainly using a narrow zustand selector into a nested array/object. See §3.
- **An old save fails to load or crashes on missing fields:** check `services/saveService.ts`'s `migrate()` function — `CURRENT_SAVE_VERSION` is now 10 (v3: Stage 3 owned equipment + shelf-life tracking; v4: Stage 4's `Equipment.currentStatus`/`repairHistory`; v5: the `attractions` array + `Customer.attractionAffinity`; v6: Stage 5 `bills`/`insolvency`; v7: `policies` and attraction prices; v8: equipment `spaceUnits`/`tier`; v9: Stage 6's `reputation`/`reviews`/`activePromotions`; v10: Real Estate/Neighborhoods' flat→per-property restructuring — wraps every pre-v10 top-level field into a single `OwnedPropertyState` and sets `properties: [wrapped]`/`activePropertyId`, splitting `bills` into loan-kind, which stays on `state.bills`, vs. everything else, which moves onto the wrapped property). `migrate()`'s pre-v10 branches deliberately keep `state` loosely typed (not `GameState`) since they mutate the old flat shape — only the v10 branch (and the final `return state as GameState`) produces the current, strictly-typed shape. If you change `GameState`'s (or any nested type's) shape again, bump the version and add another `envelope.version < N` branch rather than assuming every save was created after your change — `tests/saveService.test.ts` has a template for writing a synthetic old-version envelope directly into `localStorage` to test a migration branch.
- **A customer at the pool table gets yanked back into normal seating/ordering flow unexpectedly:** check `employeeAI.ts`'s `handleTakeOrderComplete`/`handleDeliverItemComplete` for the `using_attraction`/`waiting_for_attraction` status guards (see §3) — if you're adding a new place that sets `customer.status` based on order/task completion, it probably needs the same guard.
